# 公开与私有状态

Aleo 通过独特的状态管理机制，在区块链上实现了公开与私有数据的灵活共存。Aleo 提供了两种主要的状态存储方式：**Mappings（映射）** 和 **Records（记录）**，分别用于管理公开状态和私有状态。

## Mappings（映射）

Mappings 是 Aleo 中用于存储**公开状态**的键值对数据结构，数据直接存储在链上，可被任何人读取。

- **公开透明**：所有映射数据对网络参与者可见
- **链上存储**：状态持久化保存在区块链上
- **全局可访问**：任何程序或用户都可以查询映射中的数据

### 示例

在 `credits.aleo` 核心程序中，`account` 映射用于存储用户的公开余额：

```leo
mapping account: address => u64;
```

这意味着每个地址对应的积分余额是公开可查的，类似于传统区块链中的账户余额模型。

## Records（记录）

Records 是 Aleo 中用于管理**私有状态**的核心数据结构，其设计理念类似于比特币的 UTXO 模型，但默认情况下所有记录内容都是加密的。

### 记录的结构

每条记录包含以下关键字段：

- **所有者（Owner）**：记录所属的钱包地址
- **数据负载（Data Payload）**：加密的应用状态数据
- **序列号随机数（Serial Number Nonce）**：确保每条记录唯一且不可重放

### 核心特性

- **默认加密**：记录内容在链上加密存储，保护数据隐私
- **所有者消费**：只有记录的所有者才能消费（花费）该记录
- **不可重放**：通过序列号机制防止记录被重复消费

## 记录模型 vs 账户模型

| 特性 | 记录模型（Aleo） | 账户模型（以太坊等） |
|------|----------------|-------------------|
| 状态存储 | 加密分散在记录中 | 公开存储在账户中 |
| 数据可见性 | 默认私有，仅所有者可见 | 默认公开，所有人可见 |
| 隐私保护 | 应用状态和数据所有者均被加密 | 无内置隐私保护 |
| 并行性 | 天然支持并行处理 | 受限于账户顺序执行 |

记录模型在隐私保护方面具有显著优势：不仅应用的具体状态数据被加密，连数据的所有者身份也得到了保护，这对于构建保护用户隐私的去中心化应用至关重要。

## 状态更新机制

在 Aleo 中，状态的更新遵循**消费-产生**模式：

1. **消费旧记录**：用户消费（花费）已有的旧记录，使其失效
2. **产生新记录**：根据程序逻辑生成新的记录，反映更新后的状态
3. **零知识证明**：整个状态转换过程在链下生成零知识证明，链上验证证明的有效性

这种机制确保了：
- 状态转换的正确性可通过密码学证明验证
- 具体的计算细节和状态数据保持私密
- 网络验证者无需重新执行计算即可确认交易有效
