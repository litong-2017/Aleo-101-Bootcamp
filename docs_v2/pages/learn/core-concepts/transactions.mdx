# 交易

Aleo 交易是触发网络状态转换的基本操作单元。与其他区块链不同，Aleo 交易在**链下生成零知识证明**，然后在链上验证证明的有效性，从而实现计算的隐私保护。

## 交易结构

### 状态转换（Transitions）

每个 Aleo 交易包含一个或多个**状态转换（Transitions）**，每个 Transition 代表一次程序函数的执行：

- 每个交易最多可包含 **32 个 transitions**
- 其中一个 transition 预留用于支付**交易费**
- 剩余的 31 个 transitions 可用于执行程序逻辑

### 交易类型

Aleo 网络支持两种主要交易类型：

#### 1. 部署交易（Deployment Transaction）

用于将新程序部署到 Aleo 网络：

- 包含编译后的程序字节码
- 包含程序的零知识证明验证密钥
- 一次性操作，部署后程序永久可用

#### 2. 执行交易（Execution Transaction）

用于执行已部署程序的函数：

- 指定要调用的程序和函数
- 包含函数输入参数
- 包含执行结果的零知识证明

## 交易费用

Aleo 交易需要支付网络费用，费用结构如下：

| 操作类型 | 费用范围 |
|---------|---------|
| 基础费用 | 最低 **0.005 Aleo Credits** |
| 复杂操作 | 最高可达 **32 Aleo Credits** |

费用因素：
- 交易包含的 transitions 数量
- 零知识证明的复杂度
- 网络拥堵情况

## 零知识证明机制

Aleo 交易的核心创新在于其零知识证明机制：

### 链下证明生成

1. **执行计算**：用户在本地执行程序函数
2. **生成证明**：使用零知识证明系统（zkSNARKs）生成证明
3. **打包交易**：将证明和必要的公开输入打包为交易

### 链上证明验证

1. **接收交易**：网络验证者接收待处理交易
2. **验证证明**：验证者检查零知识证明的有效性
3. **确认状态**：验证通过后，更新链上状态

### 核心优势

- **隐私保护**：验证者无需知道具体的计算输入和中间状态
- **效率提升**：验证证明比重新执行计算快数个数量级
- **可扩展性**：复杂的计算可以在链下完成，链上仅需轻量验证

## 交易生命周期

```
用户发起交易
    │
    ▼
本地执行程序，生成零知识证明
    │
    ▼
广播交易到网络
    │
    ▼
验证者验证零知识证明
    │
    ▼
证明有效 → 更新链上状态
证明无效 → 拒绝交易
```

这种设计使得 Aleo 能够在保护用户隐私的同时，确保所有状态转换的正确性和可验证性。
